## 1. 背景与动机

### 为什么要重构

XX-Net 当前代码库存在三层结构性问题：

1. **历史包袱过重** — 项目起源于 Python 2 时代，虽已迁移至 Python 3.13，但保留了大量 Python 2 兼容代码（`six.py` 998 行、`py3_compat.py`、15 处 `sys.version_info` 分支检查），这些死代码占约 1,500 行，增加了 30%+ 的认知负担。

2. **依赖管理失控** — 核心库以源码捆绑方式内嵌于 `lib/noarch/`（hyper 0.5.0、pyasn1 0.4.3、tlslite 0.8.0-alpha43 等），多个捆绑库已严重过时或停止维护。其中 `pyasn1` 0.4.3 的安全缺陷直接影响证书验证链路的可信度。

3. **架构无法演进** — 49+ 个原始 `threading.Thread`、50+ 处全局可变状态、132 处裸 `except:`、自写的 HTTP 客户端/服务端共 1,347 行 — 这些使得任何性能优化或功能扩展都异常困难。

### 重构目标

| 目标 | 度量标准 |
|------|----------|
| 消除 Python 2 遗留 | 删除所有 `six`、`py3_compat`、`sys.version_info[0]==2` 分支 |
| 依赖安全可控 | 所有第三方库通过 pip 管理，版本可追踪、可更新 |
| 架构可演进 | 全局状态封装为类、异常处理规范化、核心模块类型注解 |
| 网络栈现代化 | 自写 HTTP 替换为标准库或成熟第三方库 |
| 可异步化 | 第四阶段可选：全面迁移至 asyncio |

### 不做的事

- 不改变用户可见功能和行为
- 不修改网络协议格式
- 不更换 Web UI 框架（第五阶段可选）
- 不更换编程语言

### 部署约束

项目采用**嵌入式 Python 运行时**方式部署（Windows）：

```
XX-Net/
├── python3/           # 嵌入式 Python 3.13
│   ├── python.exe
│   ├── python313.dll
│   └── DLLs/
│       ├── libcrypto-3.dll  (OpenSSL)
│       └── libssl-3.dll
├── code/default/      # 应用代码
└── data/              # 用户数据
```

第二阶段将捆绑库迁移至 pip 管理后，嵌入式运行时需要执行 `pip install -r requirements.txt --target python3/Lib/site-packages` 预装依赖。Linux/macOS 使用系统 Python，pip 安装无特殊处理。

**兼容性保证:** 配置文件格式（JSON）保持不变。`dataclasses` 替换 `xconfig.py` 后，`ConfigManager` 仍读写相同的 JSON 文件格式，现有用户配置无需迁移。

### 2.3 优先级排序依据

排序遵循三条规则：

1. **安全优先** — 任何涉及安全漏洞（证书验证、协议版本、过时加密库）的问题标记为 P0
2. **依赖后解耦** — 先清理依赖（阶段 1-2），再做架构变更（阶段 3-4），因为干净的依赖层是架构重构的前提
3. **低风险先行** — 同一优先级内，风险低的任务先执行，确保每个阶段结束时系统仍可运行

## 3. 总体策略

### 3.1 核心原则

| 原则 | 说明 |
|------|------|
| **渐进式** | 每个阶段独立可交付，完成后系统必须可正常运行 |
| **可回滚** | 每个阶段开始前打 git tag，失败时可回退到上一阶段 |
| **测试驱动** | 每个阶段完成后必须通过 `pytest` 全量测试 |
| **不改变行为** | 重构不改变外部可见功能和协议格式 |

### 3.2 阶段依赖关系图

```
第一阶段 ──→ 第二阶段 ──→ 第三阶段 ──→ 第四阶段（可选）
  清理遗留     依赖现代化    网络层现代化    asyncio 改造
                                              │
                                              ▼
                                         第五阶段
                                          长期优化
```

- 第一阶段 → 第二阶段：必须先删除 Python 2 代码，否则依赖替换时仍需兼容旧路径
- 第二阶段 → 第三阶段：必须先统一依赖管理，网络层替换才有明确的库可用
- 第三阶段 → 第四阶段：全局状态封装和异常规范化是 asyncio 的前置条件
- 第五阶段无前置依赖，可随时并行

### 3.3 工作量总览

| 阶段 | 预估时间 | 风险等级 | 可独立交付 |
|------|----------|----------|-----------|
| 1. 清理遗留 | 1-2 周 | 极低 | 是 |
| 2. 依赖现代化 | 2-3 周 | 低 | 是 |
| 3. 网络层现代化 | 4-8 周 | 中 | 是 |
| 4. asyncio 改造 | 4-6 个月 | 高 | 是 |
| 5. 长期优化 | 持续 | 低 | N/A |

**总计:**
- 基础现代化（阶段 1-3）: **7-13 周（约 2-3 个月）**
- 完整现代化（含 asyncio，阶段 1-4）: **6-9 个月**

#### 2.1.2 详细: hyper → h2 适配

`hyper` 和 `h2` 的 API 差异较大，需要适配层而非直接替换：

```
hyper 接口 (当前):
  conn = hyper.HTTP20Connection(host)
  conn.request('GET', url)
  resp = conn.get_response()

h2 接口 (目标):
  需要手动管理 socket + h2 协议帧
  h2.connection.H2Connection()
  conn.send_headers(stream_id, headers)
  conn.send_data(stream_id, data)
```

**适配策略:** 在 `front_base/` 下新建 `h2_adapter.py`，封装 h2 为与 `http2_connection.py` 兼容的接口，最小化改动上层代码。

## 6. 第三阶段: 网络层现代化

> **预估时间:** 4-8 周

### 6.1 HTTP 服务端替换

**当前:** `simple_http_server.py`（770 行）— 手动 epoll/select + 线程分发

**方案选择:**

| 方案 | 库 | 优点 | 缺点 | 推荐 |
|------|-----|------|------|------|
| A | stdlib `http.server` + `ThreadingMixIn` | 零新依赖 | 性能一般，无 HTTP/2 | 推荐 |
| B | `aiohttp` | 高性能，WebSocket，async | 引入 asyncio 依赖 | 不选（过早） |
| C | 优化现有 `simple_http_server.py` | 改动最小 | 未解决根本问题 | 不选 |

**推荐: 方案 A** — 保持线程模型一致，降低风险。仅替换底层，上层路由不变。

> **设计决策说明:** 服务端选 stdlib（保守）而客户端选 httpx（激进），是因为服务端仅用于本地 Web UI 管理（低并发、单用户），对性能无要求；而客户端用于 CDN 前端的数据通道连接（高并发、多连接），HTTP/2 连接池复用是刚需。两者需求不同，选型策略也不同。服务端不使用 HTTP/2 是可接受的 — 浏览器访问本地管理页面不需要 HTTP/2。

### 7.2 改造阻碍与映射

| 同步模式 | asyncio 对应 | 影响范围 |
|----------|-------------|----------|
| `threading.Thread` | `asyncio.create_task()` | 49 处 |
| `threading.Lock` | `asyncio.Lock` | ~30 处 |
| `threading.Condition` | `asyncio.Event` / `asyncio.Condition` | ~15 处 |
| `queue.Queue` | `asyncio.Queue` | ~20 处 |
| `time.sleep()` | `asyncio.sleep()` | ~40 处 |
| `select.select()` | `asyncio` 事件循环自动处理 | ~10 处 |
| `socket.recv/send` | `asyncio.StreamReader/StreamWriter` | ~50 处 |
| `ssl.wrap_socket` | `asyncio.open_connection(ssl=...)` | ~15 处 |
| 同步回调函数 | `async def` 协程 | ~25 处 |

### 7.3 推荐改造顺序

```
Step 1: HTTP 服务端 → aiohttp           ← 最独立，影响面最小
Step 2: HTTP 客户端 → httpx.AsyncClient  ← 已使用 httpx，仅切换 async
Step 3: DNS 解析 → asyncio DNS           ← 独立模块
Step 4: SOCKS 代理服务 → asyncio streams  ← 替换 proxy_handler.py
Step 5: SSL 连接管理 → asyncio SSL        ← 替换 connect_creator.py
Step 6: X-Tunnel 核心协议 → async/await   ← 最大工作量，proxy_session.py
Step 7: IP 管理器 → asyncio 任务调度       ← 替换 ip_manager.py 中的线程扫描
```

### 7.4 兼容策略

- 每个模块提供同步和异步两套接口，用 `typing.Protocol` 抽象
- 使用 `asyncio.get_event_loop().run_in_executor()` 包装尚未改造的阻塞调用
- 保留线程模式作为 fallback，通过配置开关切换

### 7.5 风险与回滚

| 风险 | 可能性 | 影响 | 应对措施 |
|------|--------|------|----------|
| asyncio 事件循环阻塞 | 中 | 整个服务冻结 | 严格区分 CPU 密集和 I/O 密集操作；CPU 密集用 executor |
| 协程间竞态条件 | 中 | 数据不一致 | 逐模块迁移，每个模块迁移后做并发压力测试 |
| SSL 与 asyncio 集成问题 | 低 | TLS 连接失败 | `asyncio.open_connection(ssl=ssl_context)` 已是成熟模式 |
| 改造中途项目需要紧急修复 | 高 | 分支管理混乱 | 在独立分支开发，main 分支始终可用 |
